Configure your tenant in 8 short steps.

This wizard generates every *.custom.* override file the SecurityInsight engines need -- tenant identity, workspace, CMDB cache, asset tagging, risk-analysis exclusions, Shodan, and more. Answers persist in your browser so you can stop and resume anytime.

Step 1 of 8 · Tenant identity

How will the SecurityInsight engines authenticate?

SecurityInsight engines authenticate to Entra and Azure once per run as a service principal. Pick a credential type, fill in the details. The wizard will emit a Layer-1 platform-defaults snippet that every engine inherits.

Engine host + bootstrap auth

Where will the SecurityInsight engines run, and how will they prove who they are when they start? This drives which credential storage options are valid below — the SI credential has to be readable by something that's already authenticated, otherwise it's a chicken-and-egg.

Bootstrap chain on Windows / Server: No Managed Identity is available, so the SI credential must be readable without first authenticating to Azure. Two valid storage options below:
• Self-signed cert + Local cert store — the cert is the auth (the SPN's certificate thumbprint identifies it to Entra). Zero KV dependency. Recommended.
• Client secret + Inline in custom.ps1 — the secret lives in your config\SecurityInsight.custom.ps1 directly. Simpler, but plaintext-on-disk (gitignored, never leaves the host).
Key Vault storage is hidden — on a Win11 / Server host with no MI, KV would itself need a separate bootstrap cert, defeating the purpose.

Bootstrap chain on Azure VM with Managed Identity: The system-assigned MI on the VM reads the SI credential from Key Vault at engine startup using its own access. Recommended: Self-signed cert + Azure Key Vault (centralised secret rotation, audit log, no plaintext on the VM). Local cert store + Inline secret also work.

Bootstrap chain on Azure Container Apps Job: The system-assigned MI on the job reads the SI credential from Key Vault at container startup. Local cert store doesn't apply (ephemeral container with no traditional Windows cert store). Recommended: Cert or Secret in Key Vault.

Step 3 of 8 · Mail / SMTP (optional)

Send risk-analysis summaries by email?

SecurityInsight can mail you a per-run summary (HTML) and attach the Excel report. Off by default — turn on only if you want the engine to send mail. Anonymous SMTP relay is fine for internal corporate mail relays; authenticated is needed for SendGrid / Brevo / Postmark / M365 SMTP AUTH.

Mail mode

Off → engine never tries to send mail. Anonymous → corporate relay (no login). Authenticated → user/password to a hosted relay.

Step 5 of 8 · Azure OpenAI (optional)

Add an AI summary to risk-analysis runs?

When enabled, RiskAnalysis sends the top findings to Azure OpenAI (your tenant, your key) and adds a Summary sheet to the Excel + a paragraph at the top of the email. Costs ~5-15 cents per run on GPT-4o-mini. Off by default; safe to enable later.

OpenAI mode

Prerequisite: you need an existing Azure OpenAI resource + a deployed model (GPT-4o-mini or GPT-4o is recommended). If you don't have one, run setup\Validate-SIOpenAI.ps1 after the wizard finishes — it provisions a PAYG instance + deployment for you.

Step 6 of 8 · Shodan attack surface (optional)

Scan your public IPs via Shodan?

When enabled, the PublicIP engine collects every Tier 0/1 asset's public IP, queries Shodan for open ports + CVE history, and lands the data in SI_PublicIP_CL for two new RA reports (open-port exposure, vulnerable-service exposure). Off by default; needs a Shodan account.

Shodan mode

Prerequisite: a Shodan API key. Free tier caps at ~100 lookups/month with no port-history; Membership ($49 one-time) lifts the cap to ~10,000/month and unlocks CVE history. SI's PublicIP engine batches lookups, so a 200-asset tenant costs ~50-100 lookups per run.

Step 7 of 8 · Output sinks + Defender XDR (optional)

Where else should output go?

By default every engine writes to Log Analytics + an Excel file. Optionally also write JSON for downstream pipelines, and link your Defender XDR workspace so RA can correlate with Defender Advanced Hunting tables.

JSON output sink

When enabled, every engine appends 'JSON' to its SI_Sinks_<Engine> array, writing newline-delimited JSON next to the Excel report. Useful for external SIEM ingest or custom analytics. Adds ~10% to runtime.

Defender XDR workspace linkage

Optionally tell SI which Log Analytics workspace your Defender XDR data is streaming to — in most tenants this is your Microsoft Sentinel workspace (if you have Sentinel deployed). Several RA reports cross-correlate (e.g. join MDE recommendations with sign-in risk events from Sentinel-ingested AAD logs). Off by default; only pick "Linked" if you have a Sentinel / Defender-XDR-streaming workspace.
